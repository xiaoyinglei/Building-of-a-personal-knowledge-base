核心模块与边界

目标：高可靠、可演进、单机能起步，未来无缝迁云

必须独立的 6 个模块

1

Ingest Pipeline

负责文件接入、OCR、解析、切块、去重、增量更新。

边界：只生产结构化知识对象，不负责回答。

2

Index Layer

管理全文索引、向量索引、元数据索引。

边界：只负责召回，不做最终推理。

3

Model Gateway

统一封装云模型、本地模型、embedding、rerank、vision。

边界：模型供应层，不知道业务上下文。

4

Retrieval Orchestrator

决定走快路径还是深路径，组合全文、向量、元数据、图谱检索。

边界：核心可靠性中枢。

5

Research Agent

负责多轮问题拆解、改写查询、跨文档比较、不确定性说明。

边界：不能直接绕过检索编排器拿答案。

6

Knowledge Layer

保存摘要、对比页、专题页、实体关系和引用索引。

边界：沉淀资产，不污染原始证据层。

云 / 本地路由

默认策略

  • 复杂推理与高质量总结：云优先
  • Embedding / rerank / OCR：优先本地或可替换
  • 敏感目录/标签：强制本地模型与本地索引
  • 云失败或超时：自动降级到本地回答链路

关键点：路由决策只在网关和策略层做，业务层不直接绑死某个模型。

GraphRAG 放置位置

正确位置：检索增强层和知识沉淀层之间。

  • 用于扩展候选证据
  • 用于发现实体关系和跨文档桥接
  • 用于生成专题页和关系视图

禁止位置：不能直接替代原文检索，不能单独主导最终回答。

为什么 Agent 不能吞掉主链路

分开的好处

  • 简单问题不被复杂规划拖慢
  • Agent 出错时不会直接破坏基础问答
  • 检索质量可以独立评估和优化

混在一起的风险

  • 每个问题都像开研究任务，延迟失控
  • 错误更难定位：是检索错还是规划错
  • 本地兜底链路会变得很重
我建议的部署演进

阶段 1：本地单机：6 个模块可以先合并成 2 到 3 个进程。

阶段 2：云主部署：Ingest、Index、Gateway、Agent、Knowledge 逐步拆开，客户端继续保持轻量。
