# 🚀 RAG 正式版「索引与检索模块」开发规范

## ❗核心原则（必须遵守）

- ❌ 不允许 Demo 级实现
- ❌ 不允许脆弱的 if-else 关键词规则
- ❌ 不允许“先跑通再说”的简化方案
- ✅ 可以接受复杂度，但必须服务于效果与扩展性
- ✅ 核心目标：**稳定产出高质量候选集（而不是仅能召回）**

---

## 🧭 模块职责边界

本模块 **只负责**：

- 索引设计
- 多路检索
- 查询理解（Query Understanding）
- 检索路由
- 多路融合
- 候选去重与整理
- 检索评估

**不负责：**

- API
- 前端
- LLM生成
- Agent流程

---

# 一、总目标

基于已完成的数据切分，构建一个**正式版检索系统**：

### 必须满足：

1. 面向真实知识库（非玩具系统）
2. 支持多路检索（非单路径）
3. 支持复杂 query
4. 可扩展（Rerank / LTR / GraphRAG / Rewrite）
5. 输出：**稳定、高质量候选集**

---

# 二、总体架构要求

## 1️⃣ 检索对象设计

- 主检索：`child chunks`
- 特殊检索：`special chunks`
- 补全：`parent chunks`

---

## 2️⃣ 必须支持的检索通道

- Dense Semantic Retrieval
- Sparse / BM25 Retrieval
- Special Retrieval（table / figure / OCR / image_summary）
- Metadata / Structure-aware Retrieval

---

## 3️⃣ Query Understanding 要求

必须具备：

- 意图识别
- 检索模式判断
- 结构感知能力
- special 内容识别

❗ 禁止：纯关键词规则

---

## 4️⃣ 融合机制要求

默认实现：

- ✅ RRF（Reciprocal Rank Fusion）

并预留：

- LTR接口
- Score calibration

---

# 三、开发任务

---

## ✅ 任务1：正式版索引结构

### 必须实现四类存储：

### 1. Child Retrieval Index
- dense + sparse 双索引
- metadata 保留

### 2. Parent Store
- parent chunk 存储
- 支持 parent_id 快速查找

### 3. Special Retrieval Index
包含：

- table
- figure
- image_summary
- ocr_region

### 4. Metadata / Structural Store
包含：

- section_path
- page
- heading
- chunk_type
- source_type

---

### ✅ 完成标准：

- 所有 chunk 可定位
- 各索引真实存在（非概念）
- 支持未来扩展

---

## ✅ 任务2：索引写入与更新

### 必须支持：

- doc级重建
- 增量写入
- 去重
- 多索引一致性写入
- 错误恢复

---

### ✅ 完成标准：

- 无脏数据
- 不重复写入
- parent-child一致

---

## ✅ 任务3：Query Understanding

### 必须输出：

```json
{
  "intent": "...",
  "needs_dense": true,
  "needs_sparse": true,
  "needs_special": false,
  "structure_constraints": {...},
  "query_type": "...",
  "confidence": 0.87
}
